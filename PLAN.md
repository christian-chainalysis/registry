# Company Design System — Implementation Plan

## Philosophy

- **Figma is the source of truth** for all visual decisions. Designers work in Figma with Token Studio. They never touch code.
- **Design tokens are the contract** between design and engineering. All visual decisions flow as CSS custom properties derived from Figma via an automated, deterministic pipeline.
- **Components are copyable defaults, not locked packages.** We use a shadcn registry. Engineers copy components into their codebase and own them locally. No npm package, no version to bump.
- **Teams use tokens and primitives but are free to compose and extend.** Two teams can have wildly different DataGrids that still look visually related because they share the same token layer.
- **No PR bottleneck.** Engineers never submit PRs to the design system repo to add product-specific functionality. They copy the base and extend locally.
- **The CI pipeline is deterministic.** No AI in the automated path. Token Studio pushes JSON, Style Dictionary transforms it, shadcn builds the registry. Predictable, auditable, no hallucination risk.
- **AI is a tool the maintainer reaches for**, not a step in automation. Use it to author new components, audit staleness, or translate Figma designs — but always human-triggered.

---

## Architecture Overview

```
Figma (Token Studio plugin)
  → tokens.json (pushed to GitHub via Token Studio sync)
    → Style Dictionary (deterministic transform)
      → globals.css (CSS custom properties)
        → shadcn build (bakes components + theme into JSON)
          → Deploy (Vercel)
            → Consuming projects (npx shadcn add @company/button)
```

## 1. Figma + Token Studio (Design Layer)

### What the designer does

- Works in Figma with the **Token Studio** plugin (free tier works; paid tier adds auto GitHub sync)
- Defines all design tokens inside Token Studio:
  - **Primitive tokens**: raw values (`blue-500: #3B82F6`, `radius-md: 6px`)
  - **Semantic tokens**: aliases (`primary → blue-500`, `background → white`)
  - **Theme sets**: Light and Dark, each with their own values for the same token names

### Token types to define

| Category | Examples | Maps to |
|----------|----------|---------|
| Color | primary, secondary, destructive, background, foreground, muted, accent, border, input, ring, card, popover | `--primary`, `--background`, etc. |
| Border Radius | sm, md, lg, xl | `--radius` and calc derivatives |
| Typography | font-family, heading sizes, body sizes, weights, line heights | Tailwind font config |
| Shadows | card elevation, dropdown shadow | `--shadow-*` vars (if needed) |
| Spacing | 1-12 scale | Tailwind spacing (usually default is fine) |

For a shadcn-based system, the **minimum viable token set** is ~30-40 tokens: the color semantic layer (15-20 light + dark pairs), radius, and font family. Spacing and typography scales can use Tailwind defaults initially.

### GitHub sync setup

- Generate a GitHub PAT with `repo` scope
- In Token Studio settings: sync provider → GitHub, repo → your registry repo, file path → `tokens/tokens.json`, branch → `main`
- On every "Push" from Token Studio, `tokens.json` is committed directly to the repo

**Free tier alternative:** Designer exports `tokens.json` manually and opens a PR. Same result, one extra step.

## 2. Registry Repo (The Engine)

### Starting point

Clone the official shadcn registry template:
```
npx degit shadcn-ui/registry-template acme-registry
```

### Target structure

```
acme-registry/
├── tokens/
│   └── tokens.json                  ← Token Studio pushes here
├── scripts/
│   └── transform-tokens.js          ← Style Dictionary config + runner
├── registry/
│   └── new-york/
│       ├── theme/
│       │   └── globals.css          ← GENERATED by Style Dictionary (never hand-edit)
│       ├── ui/
│       │   ├── button.tsx           ← all shadcn components (install with --all)
│       │   ├── input.tsx
│       │   ├── dialog.tsx
│       │   └── ... (~50 components)
│       └── blocks/                  ← composed patterns (added over time, not day one)
├── registry.json                    ← declares all items including "theme"
├── app/
│   ├── globals.css                  ← app's own CSS (imports theme for preview)
│   ├── layout.tsx
│   └── page.tsx                     ← static showcase: renders every component
├── .github/
│   └── workflows/
│       └── build.yml                ← automated pipeline
├── package.json
└── components.json
```

### Key decisions

- **Install all shadcn components** (`shadcn add --all`), not a curated subset. The cost of including unused components is zero (just a JSON file). The cost of a team needing one that's missing is a delay.
- **No blocks in v1.** Ship themed primitives only. Blocks (login form, dashboard shell, data table patterns) come later as real patterns emerge from product teams.
- **`globals.css` is a registry item.** Declare it as `registry:style` in `registry.json` so consuming teams can pull token updates with `npx shadcn add @company/theme`. Same workflow as any component.
- **The showcase page is one flat page.** Renders every component with the current theme. Its job is verification and browsing, not design. No interactive sidebar, no theme picker — Figma is where design happens.

## 3. Style Dictionary (Token Transform)

### Purpose

Transforms `tokens.json` (Token Studio format) → `globals.css` (CSS custom properties that shadcn components consume).

### Setup

```bash
npm install --save-dev style-dictionary @tokens-studio/sd-transforms
```

### The critical detail: naming mapping

Token Studio exports tokens as structured objects (`color.primary.value`). shadcn expects flat CSS vars (`--primary`). The transform script must map between these naming conventions.

**Mapping table (minimum viable):**

| Token Studio path | CSS variable | Used by |
|-------------------|-------------|---------|
| `color.background` | `--background` | `bg-background` |
| `color.foreground` | `--foreground` | `text-foreground` |
| `color.primary` | `--primary` | `bg-primary`, `text-primary` |
| `color.primary-foreground` | `--primary-foreground` | `text-primary-foreground` |
| `color.secondary` | `--secondary` | `bg-secondary` |
| `color.secondary-foreground` | `--secondary-foreground` | `text-secondary-foreground` |
| `color.muted` | `--muted` | `bg-muted` |
| `color.muted-foreground` | `--muted-foreground` | `text-muted-foreground` |
| `color.accent` | `--accent` | `bg-accent` |
| `color.accent-foreground` | `--accent-foreground` | `text-accent-foreground` |
| `color.destructive` | `--destructive` | `bg-destructive` |
| `color.border` | `--border` | `border-border` |
| `color.input` | `--input` | Input borders |
| `color.ring` | `--ring` | Focus rings |
| `color.card` | `--card` | `bg-card` |
| `color.card-foreground` | `--card-foreground` | `text-card-foreground` |
| `color.popover` | `--popover` | `bg-popover` |
| `color.popover-foreground` | `--popover-foreground` | `text-popover-foreground` |
| `borderRadius.lg` | `--radius` | All radius calc derivatives |

This mapping is the fiddliest part of the system. Get it wrong and components don't pick up tokens. Test it thoroughly.

### Running locally

```bash
node scripts/transform-tokens.js
```

Output: `registry/new-york/theme/globals.css` regenerated with current token values. This file is a generated artifact — never hand-edit it.

## 4. GitHub Actions (Automation)

### Pipeline: `.github/workflows/build.yml`

Triggers on push to `main` when `tokens/tokens.json` or `registry/**` changes.

Steps:
1. `node scripts/transform-tokens.js` — regenerate globals.css from tokens
2. `npx shadcn build` — regenerate all `public/r/*.json` registry files
3. Commit generated files back to repo (if any changed)
4. Deploy to Vercel (auto via Vercel GitHub integration)

**Result:** Designer saves in Figma → Token Studio commits → CI transforms and builds → registry deployed. Zero engineer involvement for token changes.

This pipeline is the entire system. If it works, everything works. If it breaks, nothing works.

## 5. Hosting + Auth

### Hosting

Deploy to Vercel. Each push to `main` triggers deployment automatically.

**URLs after deploy:**
```
https://registry.company.com/r/button.json
https://registry.company.com/r/theme.json
https://registry.company.com/r/dialog.json
```

### Authentication

Add Next.js middleware that validates a Bearer token on all `/r/*` routes:

```typescript
// middleware.ts
if (!isValidToken(request.headers.get("authorization"))) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 })
}
```

Consuming projects configure auth in `components.json`:

```json
{
  "registries": {
    "@company": {
      "url": "https://registry.company.com/r/{name}.json",
      "headers": {
        "Authorization": "Bearer ${COMPANY_REGISTRY_TOKEN}"
      }
    }
  }
}
```

Engineers set `COMPANY_REGISTRY_TOKEN` in `.env.local`. The token is shared internally, not committed to repos.

## 6. Consuming Projects

### One-time setup

1. `npx shadcn init` (if not already)
2. Add company registry to `components.json` (see auth section above)
3. Set `COMPANY_REGISTRY_TOKEN` in `.env.local` and CI secrets

### Day-to-day

```bash
# Add a component
npx shadcn add @company/button

# Get latest theme tokens
npx shadcn add @company/theme
```

Files land in the project, committed to git, owned by the team. They can modify freely.

### Rules for consuming teams

- **Never hard-code raw color values.** Always use CSS vars (`var(--primary)`) or Tailwind classes (`bg-primary`). Enforce with stylelint.
- **Never hand-edit globals.css.** It's a generated file. If you need overrides, add them in a separate CSS file that loads after globals.
- **Token updates are opt-in.** Re-run `shadcn add @company/theme` when you want the latest. No forced upgrades.

## 7. Versioning Strategy

**There is no formal versioning.** This is intentional — it keeps the system low-ceremony.

- Git history is the version history
- The registry URL always serves latest
- Consuming teams snapshot at the point they pull

**For significant milestones** (major token rename, breaking component restructure):
- Tag in git: `git tag v2.0.0`
- Announce in Slack with a changelog
- Teams update on their own schedule

**Branch strategy:**
- `main` → production registry (`registry.company.com`)
- Optionally `staging` → staging registry (`registry-staging.company.com`) for previewing changes before they go live

---

## 8. Known Tradeoffs

These are intentional costs of the "no bottleneck" philosophy. State them upfront so teams don't discover them by surprise:

- **No automatic staleness detection.** A team won't know their button is 6 months old unless someone tells them.
- **No merge strategy for re-pulls.** If a team customized `button.tsx` and then re-pulls, their changes are overwritten. They reconcile via git diff.
- **No dependency graph for consumers.** If `button.tsx` changes and `dialog.tsx` uses button, there's no notification to re-pull dialog.
- **No forced upgrades.** A team can run on stale tokens indefinitely. Visual inconsistency across products is possible.
- **Token naming mapping is manual.** The Style Dictionary transform must be kept in sync with shadcn's expected CSS var names. If shadcn adds a new token, someone must update the mapping.

## 9. Implementation Phases

### Phase 1: Pipeline (one afternoon)
- Clone registry template
- `shadcn add --all` to install all components
- Set up Style Dictionary with token mapping
- Create a hand-written `tokens.json` to test the pipeline
- Verify end-to-end: transform → build → serve → `shadcn add` from localhost

### Phase 2: Figma Connection (next day)
- Designer installs Token Studio
- Defines initial token set (colors, radius, font — the shadcn essentials)
- Configures GitHub sync to registry repo
- First real push: tokens.json lands, transform runs, registry rebuilds
- Verify: consuming project gets correctly themed components

### Phase 3: Deploy + Auth (that week)
- Deploy to Vercel
- Add auth middleware
- Set up `components.json` in one consuming project as proof of concept
- Build the static showcase page
- Share with team, gather feedback

### Phase 4: Mature (ongoing)
- Add blocks as patterns emerge from product teams
- Add stylelint rules to enforce token usage in consuming projects
- Build a staleness audit script (diff local copies against registry)
- Explore Figma MCP for AI-assisted component authoring (human-triggered, not automated)
